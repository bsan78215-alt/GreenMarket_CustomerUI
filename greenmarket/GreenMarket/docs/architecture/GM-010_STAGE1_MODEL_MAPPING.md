# GM-010. Stage 1 Model Mapping

**Статус:** Approved

## 1. Цель

Настоящий документ определяет соответствие между моделями предметной области GreenMarket Stage 1 и пользовательскими экранами Customer UI.
Документ обеспечивает архитектурную трассируемость между предметной областью, пользовательскими сценариями и архитектурой интерфейса.
Документ является развитием архитектурных принципов, определённых в PI-015. Domain Models, Runtime Models and UI Models.
После утверждения PR-009 Seller Profile настоящий документ использует актуальную модель предметной области, в которой публичная информация о продавце выделена в самостоятельную доменную сущность Seller Profile.

## 2. Domain Models Stage 1

К моделям предметной области Stage 1 относятся:

- Seller Profile
- ProductGroup
- Product
- SellerProduct
- SellerProductPhoto
- CatalogPublication

Дополнительно используется связанная доменная сущность:
- Market (через связь marketId из Seller Profile).

Указанные модели описывают цифровой каталог GreenMarket.
Публичная информация о продавце относится к модели Seller Profile.
Техническая сущность Seller является учётной записью платформы и не используется как публичная модель Customer UI.
Stage 1 не включает модели оформления заказа, корзины, оплаты, доставки и иных транзакционных процессов.

## 3. Runtime Models Stage 1

Platform Interface поддерживает использование Runtime Models как самостоятельного архитектурного слоя.
Однако продукт GreenMarket Stage 1 не определяет обязательных Runtime-моделей предметной области.
При необходимости реализация может использовать временные технические модели пользовательского интерфейса:
- не являются частью предметной области;
- не определяют бизнес-логику;
- не входят в продуктовую модель Stage 1;
- существуют только в рамках пользовательской сессии.

Использование подобных моделей является внутренней реализацией интерфейса и не влияет на архитектуру Stage 1.

## 4. Структура Customer UI Stage 1

### 4.1. Основные экраны
Основные точки входа Customer UI. Переход между ними не изменяет предметную модель, а предоставляет разные способы просмотра каталога.

| Экран | Архитектурное основание |
|---|---|
| Map | Seller Profile |
| Catalog | Product, ProductGroup, SellerProduct, Seller Profile |
| Seller List | Seller Profile |

### 4.2. Производные экраны
Открываются из основных экранов и отображают детали выбранных сущностей. Не являются самостоятельными точками входа.

| Экран | Архитектурное основание |
|---|---|
| Seller Card | Seller Profile |
| Product Card | Product, SellerProduct, Seller Profile |
| Search | Product, ProductGroup, SellerProduct, Seller Profile |

## 5. Platform Containers

Bottom Sheet не является самостоятельным бизнес-экраном.
Он представляет собой платформенный контейнер отображения и относится к Platform Interface.
Platform Containers не отображают предметную область напрямую и не требуют собственной Domain Model.
Они могут использоваться для отображения данных существующих Domain Models или Runtime Models, но сами не являются бизнес-сущностями.

## 6. Правило добавления экранов

Каждый новый экран обязан иметь явное архитектурное основание.
Допускаются следующие варианты:
- основной экран Customer UI;
- производный экран Customer UI;
- Platform Container / Platform Screen.

При этом каждый бизнес-экран должен иметь явную связь с утверждёнными моделями предметной области либо с утверждённой Runtime Model.
Экран, не имеющий архитектурного основания, не может быть включён в состав Stage.

## 7. Правило перехода между Stage

При расширении функциональности продукта (добавлении корзины, оформления покупки, оплаты, доставки) сначала должны быть определены соответствующие изменения предметной модели и архитектуры.
После утверждения новой продуктовой модели обновляются:
- PRD;
- Domain Models;
- настоящий документ;
- пользовательские сценарии;
- состав экранов Customer UI.

Только после этого допускается проектирование новых экранов.

## 8. Архитектурный принцип

Для GreenMarket используется следующий порядок проектирования:
1. Определяются границы Stage в PRD.
2. Формируется предметная модель Stage.
3. Определяются основные пользовательские сценарии.
4. Формируется состав основных экранов Customer UI.
5. Проектируются производные экраны.
6. Выполняется реализация пользовательского интерфейса.

Customer UI строится исключительно поверх утверждённых публичных моделей предметной области.
Технические сущности платформы не являются архитектурным основанием пользовательских экранов.
Для информации о продавце такой публичной моделью является Seller Profile, связанная с технической сущностью Seller отношением один-к-одному.
Архитектура пользовательского интерфейса должна следовать утверждённой продуктовой модели и не должна опережать её.
