# GM-UX-008. CatalogScreen — Требования на реализацию Stage 1

**Статус:** Draft
**Версия:** 2.0
**Этап:** Stage 1

---

# 1. Назначение

CatalogScreen является главным экраном просмотра опубликованных товаров GreenMarket.

Экран предоставляет покупателю единый каталог товаров всех продавцов.

Настоящий документ определяет исключительно требования к пользовательскому интерфейсу.

Предметная модель определяется документом **GM-DOM-001**.

Доступ к данным осуществляется исключительно через Repository Contract, определённый документом **GM-DOM-002**.

---

# 2. Цель

Позволить покупателю: просматривать опубликованные товары; переходить к карточке товара; переходить к карточке продавца; выполнять поиск товаров; выбирать категорию товаров.

---

# 3. Место в Customer UI

**Главные экраны Stage 1:** Map; Seller List; Catalog.

**Производные экраны:** Seller Card; Seller Catalog; Product Card.

CatalogScreen является самостоятельным главным экраном. Он не заменяет SellerCatalogScreen.

---

# 4. Источник данных

Экран не использует: REST API; SQL; DTO; локальную БД. Единственным источником данных является Repository Contract.

---

# 5. Отображаемые данные

Каждая карточка товара отображает: фотографию; название товара; единицу измерения; цену; название продавца; рейтинг продавца; расстояние до продавца (если доступно). Экран не отображает внутренние поля Domain Model.

---

# 6. Структура экрана

**Верхняя панель** — строка поиска; кнопка возврата (если используется навигацией); название экрана.

**Категории** — горизонтальный список категорий (источник: ProductGroup). Выбор категории обновляет каталог.

**Каталог** — вертикальный список карточек (источник: CatalogPublication).

---

# 7. Карточка товара

Изображение; название товара; продавца; стоимость; единицу измерения; рейтинг продавца; расстояние.

---

# 8. Действия

* **OPEN_PRODUCT** — открывает Product Card.
* **OPEN_SELLER** — открывает Seller Card.
* **SEARCH** — фильтрация каталога по поисковой строке.
* **SELECT_CATEGORY** — фильтрация по категории.

---

# 9. Не входит в Stage 1

Корзина; оформление заказа; оформление доставки; оплата; избранное; рекомендации; акции; отзывы; сортировка; фильтры; сравнение товаров.

---

# 10. Архитектурные ограничения

Запрещается: создавать новые Domain Models; изменять GM-DOM-001; изменять Repository Contract; обращаться к Backend напрямую; создавать собственные модели каталога; изменять Platform Core без отдельного архитектурного решения.

Все данные должны поступать через Repository.

---

# 11. Навигация

Допустимые переходы: Map → Catalog; Seller List → Catalog; Catalog → Product Card; Catalog → Seller Card; Seller Card → Seller Catalog; Product Card → Seller Card.

---

# 12. Производительность

Экран должен поддерживать: ленивую загрузку списка; переиспользование карточек; обновление каталога без пересоздания экрана. Конкретный механизм реализации определяется Platform Core.

---

# 13. Acceptance Criteria

* создан самостоятельный модуль `catalog`;
* реализован `CatalogScreen`;
* реализован `CatalogBuilder`;
* реализован `CatalogAdapter`;
* реализован `CatalogViewModel`;
* экран использует исключительно Repository Contract;
* отсутствуют собственные Domain Models;
* отсутствуют изменения Platform Core;
* отсутствуют изменения GM-DOM-001;
* отсутствуют изменения GM-DOM-002;
* SellerCatalogScreen продолжает работать независимо от CatalogScreen;
* пользователь может открыть карточку товара;
* пользователь может открыть карточку продавца;
* пользователь может искать товары;
* пользователь может выбирать категории товаров.
