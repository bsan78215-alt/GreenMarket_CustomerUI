## Назначение Design System

Design System GreenMarket представляет собой единый набор принципов, правил, токенов, компонентов и шаблонов, используемых при проектировании пользовательских интерфейсов GreenMarket.

Design System обеспечивает:

- единый визуальный язык продукта;
- единообразие пользовательского опыта;
- повторное использование компонентов;
- ускорение разработки;
- снижение стоимости сопровождения;
- масштабируемость интерфейсов;
- предсказуемость развития продукта.

## Цели

### Единообразие

Все интерфейсы GreenMarket должны использовать единые визуальные правила.

### Масштабируемость

Добавление новых экранов не должно требовать создания новой визуальной системы.

### Повторное использование

Максимальное количество компонентов должно использоваться повторно.

### Независимость от реализации

Design System определяет внешний вид и правила проектирования независимо от конкретной технологии реализации.

Документация не зависит от: React, Compose, Flutter, Web, iOS, Android.

### Долговременное сопровождение

Документация должна оставаться актуальной в течение всего жизненного цикла продукта.

## Основные принципы

Design System строится на следующих принципах:

- Product First
- Content First
- Consistency
- Reuse First
- Accessibility by Default
- Mobile First
- Scalability
- Simplicity

Полное описание принципов приведено в документе [DS-001-Design-Concept.md](./DS-001-Design-Concept.md).

## Структура документации

Документация состоит из следующих разделов.

**DS-001 Design Concept** — определяет философию, цели и принципы Design System.

**DS-002 Design Tokens** — определяет базовые дизайн-токены: Color, Typography, Spacing, Radius, Elevation, Motion Tokens, Icon Sizes.

**DS-003 Foundations** — определяет правила применения токенов: Grid, Layout, Accessibility, Themes, Motion, Iconography, Images, Empty States.

**DS-004 Components** — определяет все базовые UI-компоненты: Buttons, Cards, Chips, Inputs, Lists, Dialogs, Badges, Navigation Items и т.д.

**DS-005 Patterns** — определяет типовые пользовательские паттерны: Bottom Sheet, Seller Card, Product Card, Catalog, Purchase Options, Navigation, Search, Filters, Map и т.д.

**DS-006 Screen Templates** — определяет композицию экранов Customer UI. Каждый шаблон описывает структуру конкретного экрана.

**DS-007 Brand Guidelines** — определяет правила использования бренда: Voice, Tone, Photography, Illustrations, Content Guidelines.

## Правила разработки документации

Каждый документ должен:

- быть независимым от реализации.

## Версионирование

Используется семантическое версионирование.

- **Major** — изменения, несовместимые с предыдущей версией.
- **Minor** — добавление новых возможностей без нарушения совместимости.
- **Patch** — исправления ошибок и уточнения без изменения поведения.

Пример: 1.0.0 → 1.1.0 → 1.1.2 → 2.0.0.

## Правила внесения изменений

Любое изменение должно:

1. иметь обоснование;
2. не противоречить документам более высокого уровня;
3. сохранять совместимость, если не утверждено обратное;
4. сопровождаться обновлением связанных документов при необходимости.

Изменения в DS-001 требуют пересмотра зависимых документов.

Изменения в DS-002 могут потребовать обновления DS-003, DS-004 и DS-005.

## Критерии готовности документа

Документ считается завершённым, если:

- определена область ответственности;
- отсутствуют противоречия с документами более высокого уровня;
- все обязательные разделы заполнены;
- определены связанные документы;
- документ прошёл ревью и утверждение.
