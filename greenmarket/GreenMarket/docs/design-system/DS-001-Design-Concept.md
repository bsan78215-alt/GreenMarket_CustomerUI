## Назначение документа

Настоящий документ определяет общую дизайн-концепцию (Design Concept) пользовательского интерфейса GreenMarket Customer UI.

Документ является первым нормативным документом серии Design System и задаёт фундаментальные принципы проектирования пользовательского интерфейса.

Все последующие документы Design System (DS-002–DS-007) должны соответствовать настоящему документу и не могут ему противоречить.

## Цель

Основная цель Design System — создание единого визуального языка GreenMarket.

Design System должна позволять:

- проектировать новые экраны без повторного обсуждения базовых визуальных решений;
- обеспечить единообразие пользовательского опыта;
- сократить стоимость разработки;
- повысить повторное использование компонентов;
- обеспечить масштабирование продукта без деградации интерфейса.

Результатом работы является не набор экранов, а система правил, по которым проектируются любые будущие интерфейсы GreenMarket.

## Область действия

Настоящий документ распространяется на весь Customer UI Stage 1.

В дальнейшем его положения используются при разработке Stage 2, Stage 3 и последующих версий продукта, если отдельными документами не определено иное.

## Архитектурные ограничения

Архитектура Customer UI считается утверждённой и не является предметом обсуждения в рамках разработки Design System.

Design System обязана использовать существующую архитектуру без её изменения.

Каждый экран Customer UI собирается исключительно посредством существующей архитектурной цепочки:

```
ViewModel → Builder → Adapter → ContentBlock → Bottom Sheet
```

Design System не имеет права изменять:

- архитектурные контракты;
- жизненный цикл экранов;
- модель данных;
- способы взаимодействия компонентов;
- Platform Core.

Если предлагаемое визуальное решение требует изменения архитектуры, такое решение считается выходящим за рамки настоящего документа и оформляется отдельным архитектурным предложением.

## Видение продукта

GreenMarket не является классическим маркетплейсом.

GreenMarket также не является интернет-магазином.

Основная задача продукта — помочь пользователю покупать товары на обычном рынке быстрее, удобнее и прозрачнее.

Следовательно, интерфейс должен создавать ощущение:

- настоящего рынка;
- живого общения;
- доверия;
- свежести продуктов;
- простоты;
- скорости;
- лёгкости.

При этом интерфейс не должен выглядеть как корпоративная информационная система.

## Визуальная идентичность

Желаемые ассоциации:

- натуральность;
- спокойствие;
- современность;
- открытость;
- чистота;
- аккуратность;
- свежесть;
- дружелюбие;
- локальная торговля.

Недопустимые ассоциации:

- банковские приложения;
- CRM;
- ERP;
- государственные сервисы;
- игровые интерфейсы;
- перегруженные маркетплейсы;
- рекламные каталоги.

Каждое визуальное решение должно усиливать желаемые ассоциации и не создавать нежелательные.

## Главный принцип Design System

Дизайн подчиняется продукту.

Любое визуальное решение должно улучшать выполнение пользовательского сценария.

Если декоративный элемент не повышает удобство использования, он не должен включаться в Design System.

При конфликте между эстетикой и удобством приоритет всегда отдаётся удобству.

## Базовые принципы проектирования

Настоящий раздел определяет фундаментальные принципы, которыми необходимо руководствоваться при разработке любых экранов Customer UI.

Эти принципы являются обязательными для всех компонентов Design System.

### Content First

Информация всегда имеет приоритет над декоративным оформлением.

Основной задачей интерфейса является максимально быстрое донесение информации до пользователя.

Не допускается использование декоративных элементов, ухудшающих читаемость или отвлекающих внимание.

Следствия:

- фотографии товара важнее декоративной графики;
- текст важнее цветовых эффектов;
- композиция должна помогать чтению, а не усложнять его.

### White Space First

Основным способом разделения информации является свободное пространство.

Предпочтительно использовать:

- интервалы;
- группировку;
- визуальную иерархию.

Использование большого количества линий, рамок и разделителей допускается только при невозможности решить задачу посредством композиции.

### Large Touch Targets

Интерфейс ориентирован прежде всего на использование одной рукой.

Все интерактивные элементы должны иметь достаточную область касания.

Минимальные требования:

- минимальная область касания — 48 dp;
- основные кнопки должны быть легко достижимы большим пальцем;
- часто используемые действия располагаются в нижней части экрана.

### Progressive Disclosure

Пользователь должен видеть только ту информацию, которая необходима на текущем этапе сценария.

Дополнительные сведения раскрываются постепенно.

Для раскрытия информации преимущественно используется Bottom Sheet.

Не допускается перегружать экран редко используемой информацией.

### One Primary Action

На каждом экране существует только одно главное действие.

Главное действие должно быть очевидным визуально и не конкурировать с второстепенными элементами.

Если пользователь затрудняется определить следующее действие, дизайн считается неудачным.

### Consistency

Одинаковые сущности отображаются одинаково.

Одинаковые действия выполняются одинаковым способом.

Изменение внешнего вида компонента допускается только при изменении его смысла.

Недопустимо существование нескольких вариантов одного и того же компонента без объективной причины.

### Motion With Purpose

Анимация используется только тогда, когда она улучшает понимание происходящего.

Допустимые цели анимации:

- объяснение изменения состояния;
- сохранение визуального контекста;
- облегчение восприятия переходов;
- подтверждение действий пользователя.

Недопустимо использование декоративной анимации, не несущей функциональной пользы.

### Sunlight First

Customer UI проектируется прежде всего для использования вне помещений.

Интерфейс должен сохранять читаемость:

- при ярком солнечном освещении;
- при низком качестве экрана устройства;
- при загрязнённом защитном стекле;
- при быстром взгляде на экран.

Высокая читаемость имеет приоритет над визуальной выразительностью.

### Fast Perception

Пользователь должен понимать содержимое любого экрана за 2–3 секунды.

Для достижения этой цели используются:

- чёткая визуальная иерархия;
- ограниченное количество акцентов;
- предсказуемое расположение элементов;
- единообразие компонентов.

Если понимание структуры экрана требует длительного изучения, такой экран считается не соответствующим требованиям Design System.

## Продуктовые принципы GreenMarket

В отличие от универсальных UX-принципов, настоящий раздел отражает особенности именно GreenMarket.

Эти принципы определяют характер продукта и должны находить отражение в последующих документах Design System.

### Living Marketplace

Интерфейс должен передавать ощущение живого рынка, а не безличного каталога.

Каждый продавец должен сохранять свою индивидуальность.

Следствия. Обязательными элементами карточки продавца являются:

- реальная фотография (при наличии);
- название;
- рейтинг;
- отличительные характеристики.

Индивидуальность продавцов достигается содержанием карточек, а не использованием различных цветовых схем или декоративных эффектов.

### Freshness First

Для покупателя важно понимать актуальность информации.

Интерфейс должен помогать быстро оценить:

- актуальность цены;
- наличие товара;
- свежесть данных.

Следствия: все механизмы отображения актуальности должны быть единообразными во всей системе. Не допускается скрывать информацию об устаревании данных.

### Local Trust

Доверие пользователя формируется реальными данными, а не декоративным оформлением.

Источниками доверия являются:

- реальные фотографии;
- рейтинг;
- расстояние;
- информация о продавце;
- подтверждённые характеристики.

Дизайн не должен создавать ложное ощущение надёжности за счёт визуальных эффектов.

### Compare Naturally

Основная задача интерфейса — облегчить сравнение предложений.

Информация одного типа должна располагаться в одинаковых местах на всех карточках и экранах.

Сравнение не должно требовать открытия большого количества дополнительных экранов.

Визуальная структура должна позволять пользователю быстро сопоставлять предложения разных продавцов.

## Accessibility

Доступность является обязательным требованием Design System и распространяется на все компоненты без исключения.

GreenMarket следует требованиям WCAG AA как определено Material Design 3 (https://m3.material.io/foundations/accessible-design); если настоящим документом не определено иное, применяются рекомендации MD3.

### Приоритеты GreenMarket

Для GreenMarket наибольшее значение имеют следующие требования, дополняющие базовый MD3 Accessibility:

**Высокая читаемость.** Интерфейс должен сохранять читаемость при ярком солнечном освещении, на бюджетных устройствах, при использовании защитных стёкол и при быстром просмотре.

**Простота восприятия.** Основные пользовательские действия должны быть понятны без предварительного обучения.

**Предсказуемость.** Одинаковые действия имеют одинаковое визуальное представление.

**Независимость от цвета.** Если цвет передаёт состояние, оно дублируется текстом, иконкой, бейджем или другим визуальным способом.

## Graceful Degradation

### Общие положения

Design System должна корректно работать даже при неполных или отсутствующих данных.

Компоненты обязаны иметь предсказуемое поведение в условиях деградации.

### Обязательные состояния компонентов

Каждый компонент Design System должен поддерживать следующие состояния.

**Loading.** Данные загружаются. Используются Skeleton Placeholder или аналогичные механизмы.

**Skeleton.** Отображается структура будущего содержимого. Не допускается резкое изменение композиции после загрузки.

**Empty.** Данные отсутствуют. Пользователь получает понятное объяснение причины.

**Partial.** Доступна только часть информации. Интерфейс продолжает оставаться работоспособным.

**Error.** Возникла ошибка получения данных. Пользователь получает понятное сообщение и возможность повторить действие.

**Offline.** Сетевое соединение отсутствует. Должны отображаться:

- последнее известное состояние (если доступно);
- информация об отсутствии соединения;
- возможность повторной синхронизации.

**Disabled.** Элемент временно недоступен. Причина недоступности должна быть понятна пользователю.

## Проверка продуктовых гипотез

### Общие положения

Ряд решений Design Concept основан на продуктовых гипотезах и требует подтверждения в ходе пилотной эксплуатации.

До получения результатов пилота данные решения считаются предварительно утверждёнными.

### Основная гипотеза

Customer UI должен быть одинаково удобен пользователям с различным уровнем цифровой подготовки.

### Метрики проверки

В ходе пилотной эксплуатации рекомендуется оценивать:

- процент пользователей, оформивших первый заказ без помощи;
- среднее время оформления первого заказа;
- количество возвратов назад;
- количество отменённых действий;
- количество обращений за помощью;
- субъективную оценку удобства использования;
- различия между возрастными группами пользователей (при наличии таких данных).

### План действий

Если пилот покажет существенные проблемы восприятия интерфейса, допускается изменение следующих параметров Design System:

- размеры шрифтов;
- интервалы;
- плотность информации;
- размеры элементов управления;
- контрастность;
- визуальная иерархия.

При этом не допускается изменение базовых принципов Design Concept без отдельного пересмотра настоящего документа.

## Критерии завершения документа

Design Concept считается завершённым после выполнения следующих условий. Утверждены:

- общая визуальная концепция;
- базовые UX-принципы;
- продуктовые принципы;
- критерии выбора дизайн-решений;
- требования Accessibility;
- требования Graceful Degradation;
- порядок проверки продуктовых гипотез.

После утверждения настоящего документа последующие документы Design System должны соответствовать его положениям.



## Статус документа

Настоящий документ определяет фундаментальные принципы визуального проектирования GreenMarket Customer UI.

После утверждения он считается нормативным документом Design System.

Изменение настоящего документа допускается только в одном из следующих случаев:

- изменение Product Vision;
- изменение архитектурных ограничений;
- результаты пилотной эксплуатации, свидетельствующие о необходимости пересмотра базовых принципов;
- принятие новой версии Design System.

Во всех остальных случаях изменения должны вноситься в последующие документы серии DS, не изменяя фундаментальные положения настоящей спецификации.
