# ТЗ-026. Протокол верификации ТЗ по результатам реализации

**Статус:** Draft

- ТЗ-024 — Bottom Sheet
- ТЗ-013 — Правила развития Customer UI

**Основной читатель**
- ★★★★★ React Developer
- ★★★★☆ Tech Lead / Архитектор
- ★★★☆☆ UX Designer
- ★★☆☆☆ QA

## Назначение

Настоящий документ определяет порядок проверки качества технического задания после реализации первой версии функциональности.

Цель проверки — оценить не качество разработчика, а полноту и однозначность ТЗ.

## Принцип

Хорошее ТЗ позволяет реализовать функциональность без самостоятельного принятия архитектурных решений.

Если разработчик вынужден регулярно додумывать поведение системы, это считается недостатком ТЗ.

## Объект проверки

В первой итерации проверяется:

- ТЗ-025. Карточка продавца

по результатам реализации файла:

- `BottomSheetDeclarative_3.jsx`

## Проверяемые критерии

### VR-001. Полнота

Проверяется, все ли основные элементы карточки были описаны в ТЗ.

Например:

- Header
- Purchase Coverage
- Seller Information
- Basket Products
- Other Products
- Reports
- Bottom Actions

Если разработчик добавил новый крупный блок самостоятельно, необходимо определить:

- это ошибка ТЗ;
- или лишняя функциональность.

### VR-002. Однозначность

Проверяется, приходилось ли разработчику самостоятельно принимать решения относительно:

- структуры карточки;
- порядка отображения блоков;
- поведения Bottom Sheet;
- взаимодействия с картой.

Если такие решения принимались, соответствующие требования должны быть добавлены в ТЗ.

### VR-003. Архитектурная совместимость

Проверяется соответствие архитектурным ограничениям проекта.

Не допускается:

- бизнес-логика в React;
- работа напрямую с картой;
- вычисление доступных действий в UI;
- использование `status` как источника решений.

### VR-004. Соответствие Platform Core

Проверяется использование:

- Action;
- Business Events;
- ViewModel;
- availableActions.

Все взаимодействия должны соответствовать утверждённой архитектуре Platform Core.

### VR-005. Избыточность ТЗ

Для каждого раздела ТЗ определяется:

- использован при реализации;
- оказался полезным;
- не использовался.

Неиспользуемые разделы анализируются отдельно.

### VR-006. Недостающие требования

Фиксируются все вопросы, возникшие у разработчика в процессе реализации.

Каждый вопрос должен привести к одному из решений:

- дополнить ТЗ;
- изменить архитектуру;
- оставить без изменений.

## Результат проверки

По итогам ревью для каждого требования выставляется один из статусов:

- ✔ Подтверждено реализацией.
- ⚠ Требует уточнения.
- ✖ Не подтверждено.
- ➕ Требуется новое требование.

## Правила обновления ТЗ

После завершения проверки допускается:

- уточнение существующих требований;
- добавление недостающих требований;
- удаление требований, не имеющих практической ценности.

Не допускается изменение архитектурных принципов без отдельного архитектурного решения.

## Критерий качества ТЗ

Техническое задание считается качественным, если:

- разработчик реализовал функциональность без изменения архитектуры;
- количество самостоятельных архитектурных решений минимально;
- все необходимые данные были получены из документации;
- реализация соответствует принятой модели Platform Core.

## Применение

Настоящий протокол используется для проверки всех последующих экранов GreenMarket:

- Карта.
- Каталог.
- Карточка товара.
- Purchase Option.
- Домашние запасы.
- Поиск.

После реализации каждого экрана сначала проводится ревизия ТЗ, и только затем документ получает статус Approved.

## Почему этот документ важен

До сих пор документация писалась «вперёд». Теперь проект достиг стадии, когда документация начинает проверяться реальной разработкой. Этот протокол вводит постоянный цикл улучшения:

```
ТЗ → Реализация → Ревью → Уточнение ТЗ → Approved
```

Именно такой процесс позволит постепенно превратить документацию GreenMarket в проверенную инженерную спецификацию, а не просто набор проектных решений.
