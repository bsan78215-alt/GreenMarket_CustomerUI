## Цель

Пользователь хочет быстро купить привычный набор продуктов с минимальными затратами времени.

## Предусловия

Пользователь открыл приложение. Может быть: новым; постоянным.

## Шаг 1. Открытие приложения

Отображается главный экран.

Карта уже содержит: любимых продавцов; ближайшие точки; продавцов возле сохранённых мест; продавцов вдоль регулярных маршрутов.

Bottom Sheet показывает: текущую корзину; домашние запасы; типовые корзины.

## Шаг 2. Проверка покупки

Пользователь оценивает: всё ли осталось актуальным; нужно ли что-нибудь добавить; нужно ли убрать товар.

При необходимости: меняет корзину; меняет домашние запасы; добавляет товар через поиск.

После каждого изменения итоговая покупка пересчитывается автоматически.

## Шаг 3. Подбор вариантов

Пользователь нажимает «Подобрать покупку». Система формирует несколько вариантов. Например: рекомендуемый; любимые продавцы; минимальная цена; минимальный маршрут.

## Шаг 4. Анализ вариантов

Bottom Sheet показывает найденные варианты. При выборе варианта: карта обновляется; подсвечиваются участвующие продавцы; строится маршрут.

## Шаг 5. Просмотр продавца

Пользователь открывает продавца. Видит: товары из своей покупки; отсутствующие товары; возможные замены; остальной каталог.

При необходимости: меняет товары; добавляет товары; выбирает замену.

После изменения покупка пересчитывается автоматически.

## Шаг 6. Просмотр товара

Пользователь оценивает: фотографии; качество; происхождение; наличие; цену; время последнего обновления.

При необходимости изменяет покупку.

## Шаг 7. Начало покупки

Пользователь выбирает «Начать маршрут». Карта переключается в режим сопровождения покупки.

## Шаг 8. Во время покупки

При посещении продавца пользователь может: подтвердить наличие товара; сообщить об отсутствии; подтвердить цену; сообщить об изменении цены; подтвердить качество; предложить замену.

Эти действия помогают поддерживать актуальность каталога.

## Шаг 9. Завершение покупки

После завершения система предлагает: обновить домашние запасы; сохранить изменения в корзине; отметить понравившихся продавцов.

Если пользователь работает без регистрации, предлагается зарегистрироваться для сохранения настроек и истории.

## Результат сценария

В результате выполнения сценария: покупка завершена; домашние запасы актуализированы; информация о продавцах и товарах при необходимости обновлена; история покупок сохранена (локально или в облаке).

**Комментарий из чата (переход к следующему документу):** мне кажется, после этого документа не стоит продолжать описывать пользовательские сценарии. Следующим логичным документом должен стать «Состояния интерфейса (UI State Model)». Не экраны, а именно состояния, например: Initial; Browsing; Editing Basket; Calculating Purchase; Viewing Options; Viewing Seller; Viewing Product; Shopping Route; Purchase Completed.

Это будет очень полезно и для разработчика Customer UI, и для интеграции с Platform Core. Вместо набора разрозненных экранов появится понятная модель переходов между состояниями, которая хорошо ложится на архитектуру взаимодействий.
