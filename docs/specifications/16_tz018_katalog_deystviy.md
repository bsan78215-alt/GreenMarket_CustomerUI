## Назначение

Настоящий документ определяет полный каталог действий пользователя.

Любое действие Customer UI должно соответствовать одному из действий настоящего каталога.

Запрещается создавать действия, отсутствующие в каталоге, без внесения изменений в настоящий документ.

## Принцип

Action описывает намерение пользователя, а не техническую реализацию.

Правильно: `OpenSeller`. Неправильно: `OpenSellerBottomSheet` или `NavigateSellerScreen` — UI сам решает, как отобразить результат.

## Раздел 1. Навигация

- `OpenMainScreen` — открыть главный экран.
- `OpenBasket` — открыть корзину.
- `OpenSeller` — открыть продавца.
- `OpenProduct` — открыть товар.
- `OpenPurchaseOption` — открыть вариант покупки.
- `OpenSearch` — открыть поиск.
- `OpenSettings` — открыть настройки.

## Раздел 2. Покупка

- `SelectBasket` — выбрать корзину.
- `CreateBasket` — создать корзину.
- `SaveBasket` — сохранить корзину.
- `DeleteBasket` — удалить корзину.
- `AddProduct` — добавить товар.
- `RemoveProduct` — удалить товар.
- `ChangeQuantity` — изменить количество.
- `ReplaceProduct` — заменить товар.

## Раздел 3. Домашние запасы

- `AddHomeInventory` — добавить домашний остаток.
- `RemoveHomeInventory` — удалить домашний остаток.
- `UpdateHomeInventory` — изменить домашний остаток.

## Раздел 4. Подбор покупки

- `BuildPurchaseOptions` — подобрать варианты покупки.
- `SelectPurchaseOption` — выбрать вариант покупки.
- `StartShoppingRoute` — начать маршрут.
- `FinishShoppingRoute` — завершить маршрут.

## Раздел 5. Карта

- `MoveMap` — переместить карту.
- `ZoomMap` — изменить масштаб.
- `SelectMarker` — выбрать точку.
- `ChangeMapMode` — изменить режим карты.

## Раздел 6. Поиск

- `SearchProduct` — найти товар.
- `FilterProducts` — изменить фильтр.
- `SortProducts` — изменить сортировку.

## Раздел 7. Продавец

- `AddFavoriteSeller` — добавить в любимые.
- `RemoveFavoriteSeller` — удалить из любимых.
- `BuildRouteToSeller` — построить маршрут.

## Раздел 8. Обратная связь

- `ConfirmAvailability` — подтвердить наличие.
- `ReportMissingProduct` — сообщить об отсутствии.
- `ReportPriceChange` — сообщить об изменении цены.
- `ReportQualityChange` — сообщить об изменении качества.
- `UploadProductPhoto` — добавить фотографию.

## Требования к действиям

Каждое действие должно: иметь одно понятное назначение; быть независимым от конкретного интерфейса; одинаково работать из графического интерфейса, голосового интерфейса и других каналов; порождать одно или несколько бизнес-событий.

## Связь с Platform Core

Каждое действие GreenMarket должно напрямую отображаться на Action Platform Core. Например:

| GreenMarket | Platform Core |
|---|---|
| OpenSeller | `Action(type="seller.open")` |
| SearchProduct | `Action(type="product.search")` |
| BuildPurchaseOptions | `Action(type="purchase.optimize")` |
| ReportMissingProduct | `Action(type="product.report_missing")` |

Platform Core не должна знать бизнес-смысл этих действий — она обеспечивает их передачу, обработку и журналирование.

**Комментарий из чата (переход к следующему документу):** предлагаю следующий документ сделать очень необычным. До сих пор мы описывали что пользователь может сделать. Теперь стоит описать что пользователь видит на каждом этапе — не экран, не компоненты, а именно информационные приоритеты. Например:

- Главный экран — в первую очередь: есть ли что-то интересное рядом; как быстро начать привычную покупку; что изменилось с прошлого посещения.
- Карточка продавца — в первую очередь: насколько этот продавец закрывает текущую покупку; что отсутствует; что можно заменить.
- Карточка товара — в первую очередь: стоит ли его покупать сегодня; почему система его рекомендует; насколько информация о нём свежая.

Такой документ будет полезен дизайнеру — он задаст не расположение элементов, а иерархию информации, которая определяет качество пользовательского опыта.
