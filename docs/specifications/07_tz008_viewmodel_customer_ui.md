## Назначение

ViewModel определяет все объекты, с которыми работает интерфейс покупателя.

Она является контрактом между Backend, Platform Core и Customer UI.

ViewModel не должна повторять структуру БД. Один объект ViewModel может объединять данные из нескольких таблиц.

## 1. UserContext

Описывает текущего пользователя. Содержит: авторизован ли пользователь; текущее местоположение; активное значимое место (Дом, Работа и т.п.); активный маршрут (если есть); настройки отображения карты.

## 2. PurchaseIntent

Главный объект интерфейса. Именно вокруг него строится работа приложения.

Содержит: активную корзину; итоговый список покупки; исключённые домашними запасами товары; товары, добавленные вручную; пользовательские ограничения; предпочтения.

PurchaseIntent не хранится в БД. Он вычисляется.

## 3. PurchaseOption

Описывает один вариант покупки. Например: рекомендуемый; самый дешёвый; любимые продавцы; минимальный маршрут.

Содержит: стоимость; маршрут; продавцов; покрытие корзины; отсутствующие товары.

## 4. SellerSummary

Краткая информация для карты. Используется при отображении маркеров.

Содержит: название; координаты; рейтинг; расстояние; признак «любимый»; участие в текущем варианте покупки.

## 5. SellerDetails

Полная информация карточки продавца.

Содержит: общую информацию; товары из текущей покупки; остальные товары; предлагаемые замены; информацию о качестве.

## 6. ProductSummary

Используется: в корзине; поиске; списках. Содержит минимальный набор данных.

## 7. ProductDetails

Полная карточка товара.

Содержит: фотографии; цену; наличие; происхождение; сезонность; историю обновления; рекомендации.

## 8. BasketView

Отображение активной корзины.

Содержит: товары; количество; исключённые товары; итоговое количество позиций.

## 9. HomeInventoryView

Отображение домашних запасов. Используется только UI.

## 10. MapContext

Определяет текущее состояние карты. Например, режим: обзор; подбор покупки; маршрут; просмотр продавца.

## 11. BottomSheetContext

Определяет содержимое рабочей панели. Например, сейчас открыто: корзина; варианты покупки; продавец; товар.

## Принципы ViewModel

1. **Независимость от БД** — изменение структуры базы данных не должно автоматически требовать изменения ViewModel.
2. **Независимость от API** — Backend может агрегировать данные любым способом, UI получает уже готовые объекты.
3. **Независимость от Platform Core** — Platform Core ничего не знает о Basket, Product или Seller. Она работает только с действиями и событиями. ViewModel является адаптером между предметной областью GreenMarket и Platform Core.

## Совместимость с будущим FSM Engine

> Добавлено по правкам из [`21_prompt_fsm_engine_sovmestimost.md`](21_prompt_fsm_engine_sovmestimost.md) / ТЗ-022.

ViewModel не зависит от способа реализации бизнес-логики.

Источник данных может быть:

- CRUD;
- FSM Engine;
- любая другая реализация.

Структура ViewModel при этом не изменяется.
