## Назначение

Backend предоставляет Customer UI готовые предметные объекты.

Customer UI не занимается объединением данных из нескольких запросов. Backend возвращает уже готовые модели для отображения.

## Принцип №1. UI не собирает экран самостоятельно

Плохо: `GET Seller`, `GET Catalog`, `GET Stock`, `GET Price`, `GET Reviews` — React должен объединить всё сам.

Правильно: `GET Seller Details` — Backend возвращает полностью готовую карточку продавца.

## Принцип №2. Один пользовательский сценарий — один запрос

- Открытие продавца: UI вызывает `Open Seller`, Backend возвращает `SellerDetailsView`.
- Подбор покупки: UI вызывает `Build Purchase Options`, Backend возвращает `PurchaseOptionsView`.
- Открытие товара: Backend возвращает `ProductDetailsView`.

## Принцип №3. Backend знает контекст пользователя

Например, если открыт продавец, Backend уже знает текущую корзину; домашние запасы; любимых продавцов.

Поэтому он сразу возвращает «4 из 5 товаров вашей покупки имеются». React не вычисляет это.

## Контракт главного экрана

После открытия приложения Backend возвращает `MainScreenView`. Она содержит:

- MapContext
- PurchaseIntent
- BasketSummary
- HomeInventorySummary
- SuggestedBaskets
- FavoriteSellers
- NearbySellers

Главный экран строится полностью из одного объекта.

## Контракт вариантов покупки

`PurchaseOptionsView` содержит: PurchaseIntent; Option[]; SelectedOption; Statistics.

## Контракт продавца

`SellerDetailsView` содержит: Seller; ProductsFromPurchase; MissingProducts; SuggestedReplacements; OtherProducts.

## Контракт товара

`ProductDetailsView` содержит: Product; Seller; Availability; Quality; Alternatives.

## Очень важное правило

UI никогда не должен спрашивать «А есть ли у продавца товар X?» — Backend уже обязан вернуть готовый ответ.

## Ещё более важное правило

UI вообще не должен знать, из каких таблиц получены данные. Он знает только ViewModel.

## Требование стабильности контрактов

> Добавлено по правкам из [`21_prompt_fsm_engine_sovmestimost.md`](21_prompt_fsm_engine_sovmestimost.md) / ТЗ-022.

Публичные контракты между Customer UI и Backend не должны зависеть от способа реализации бизнес-логики.

Замена CRUD на FSM Engine не должна требовать изменения API Customer UI.

**Комментарий из чата (переход к следующему документу):** я думаю, после этого документа становится понятна ещё одна вещь — нам начинает не хватать слов. Мы всё время говорим Basket, Purchase, Purchase Intent, Purchase Option, но они уже имеют совершенно разные смыслы.

Я бы следующим документом сделал не API, а глоссарий GreenMarket. Например:

| Термин | Значение |
|---|---|
| Basket | Постоянный шаблон покупок пользователя |
| Purchase Intent | Покупка, сформированная на сегодняшний день с учётом домашних запасов |
| Purchase Option | Один из рассчитанных системой способов выполнить Purchase Intent |
| Shopping Route | Последовательность посещения продавцов для выбранного Purchase Option |
| Home Inventory | Домашние запасы пользователя |
| Place | Значимое место пользователя |
| Scenario | Типовая корзина или пользовательский шаблон покупки |

Это не выглядит техническим документом, но на практике именно он предотвращает десятки недоразумений между разработчиками. Сейчас, когда проект быстро развивается, единый словарь станет не менее важным, чем API или схема БД.
