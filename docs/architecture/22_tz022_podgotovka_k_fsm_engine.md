## Назначение

Настоящий документ определяет требования к реализации GreenMarket до появления FSM Engine.

Цель — обеспечить возможность замены текущей CRUD-реализации бизнес-логики на FSM Engine без изменения архитектуры Customer UI, публичных контрактов Backend и пользовательских сценариев.

Настоящее ТЗ распространяется на Backend, Customer UI и сервисы предметной области.

## Цели

После внедрения FSM Engine не должны изменяться: Customer UI; ViewModel; User Flows; Business Rules; Action Catalog; Business Event Catalog; публичный REST API.

Допускается изменение только внутренней реализации обработки бизнес-действий.

## Архитектурный принцип

В текущей версии проекта Platform Core уже является частью архитектуры.

FSM Engine будет встроен внутрь существующей архитектуры Platform Core и заменит текущую реализацию бизнес-логики. Никакие внешние контракты при этом изменяться не должны.

## Требование 1. Все изменения выполняются через Action

Любое изменение предметной области должно инициироваться Action. Запрещается прямое изменение бизнес-сущностей из пользовательского интерфейса.

Правильно: `AddProduct`, `RemoveProduct`, `SelectPurchaseOption`, `ConfirmAvailability`.

Неправильно: изменение полей сущности напрямую; изменение состояния через UI без выполнения Action.

## Требование 2. Все изменения завершаются Business Event

После успешного выполнения Action система обязана сформировать одно или несколько Business Events.

Например: Action `AddProduct` → результат `ProductAddedToPurchase`, `PurchaseIntentChanged`.

Business Events являются частью публичной модели предметной области и должны сохранять свою семантику после внедрения FSM Engine.

## Требование 3. Запрет использования status в бизнес-логике

Поле `status` не должно использоваться как источник принятия решений.

Запрещается: `if (order.status == READY)` или `switch(order.status)`.

Бизнес-решения принимаются исключительно сервисами предметной области.

## Требование 4. Customer UI не вычисляет бизнес-логику

Customer UI не определяет самостоятельно: доступность действий; допустимость операций; последовательность переходов; ограничения.

UI отображает только данные, полученные в составе ViewModel.

## Требование 5. ViewModel независима от реализации

ViewModel является контрактом между Backend и Customer UI. Источник формирования ViewModel может быть: CRUD; FSM Engine; другая реализация.

Структура ViewModel не должна зависеть от способа реализации бизнес-логики.

## Требование 6. Стабильность Action Catalog

Каталог действий является публичным контрактом. После внедрения FSM Engine список Actions не должен изменяться. Допускается изменение только внутренней реализации их обработки.

## Требование 7. Стабильность Business Event Catalog

Business Events описывают факты предметной области. FSM Engine становится дополнительным механизмом формирования этих событий. Названия, смысл и состав Business Events изменяться не должны.

## Требование 8. Сервисы предметной области изолированы от реализации

Все сервисы должны обращаться к абстракции обработки действий. Им запрещается зависеть от конкретной реализации: CRUD; FSM Engine; иных механизмов.

Замена реализации не должна приводить к изменению публичных интерфейсов сервисов.

## Требование 9. Customer UI не зависит от FSM

Customer UI работает исключительно с: ViewModel; Action Catalog; Business Events.

Customer UI не должен знать: существует ли FSM Engine; используется ли CRUD; каким образом принимаются бизнес-решения.

## Требование 10. Критерий готовности к внедрению FSM

GreenMarket считается подготовленным к внедрению FSM Engine, если его подключение требует изменения только: внутренних обработчиков Action; внутренней реализации сервисов предметной области; конфигурации Platform Core.

При этом не требуется изменение: Customer UI; REST API; структуры ViewModel; пользовательских сценариев; бизнес-правил; каталогов Actions и Business Events.

## Критерии приёмки

При реализации каждого нового модуля необходимо подтвердить: отсутствует зависимость бизнес-логики от поля status; все изменения инициируются через Action; каждое успешное действие порождает Business Event; Customer UI не содержит бизнес-логики; ViewModel не зависит от способа реализации; замена CRUD на FSM Engine не требует изменения публичных контрактов.
