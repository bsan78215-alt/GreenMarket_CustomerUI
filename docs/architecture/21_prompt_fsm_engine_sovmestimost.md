## Промпт

### 1. README.md

Добавь после раздела с описанием архива новый раздел:

> **Архитектурное ограничение**
>
> Platform Core уже является частью архитектуры проекта.
>
> FSM Engine в текущую версию не входит и будет внедрён позже.
>
> Все документы Customer UI должны проектироваться так, чтобы переход от текущей CRUD-реализации к FSM Engine потребовал замены реализации сервисов, но не изменения: Customer UI; ViewModel; User Flows; Business Rules; Action Catalog; Event Catalog.

### 2. Документ «Каталог действий» (ТЗ-018)

Добавь в начало документа раздел:

> **Требование совместимости с будущим FSM Engine**
>
> Все Actions являются долгоживущим публичным контрактом. После внедрения FSM Engine список Actions не должен измениться. Изменению подлежит только механизм их обработки.

### 3. Документ «Каталог бизнес-событий» (ТЗ-017)

Добавь раздел:

> **Требование совместимости с будущим FSM Engine**
>
> Business Events являются фактами предметной области. После внедрения FSM Engine состав и смысл Business Events должны сохраниться. FSM Engine становится ещё одним источником этих событий, но не изменяет их семантику.

### 4. Документ «Business Rules» (ТЗ-020)

Добавь новые правила:

> **BR-013.** Бизнес-логика не должна принимать решения на основании поля `status`. Все решения должны приниматься через сервисы предметной области или политики.
>
> **BR-014.** Все изменения предметной области инициируются Action. Прямое изменение бизнес-сущностей из UI запрещено.
>
> **BR-015.** UI не должен определять доступность действий самостоятельно. UI получает список доступных действий из Backend/ViewModel.

### 7. Документ «Контракт UI ↔ Backend» (ТЗ-009)

Добавить раздел:

> **Требование стабильности контрактов**
>
> Публичные контракты между Customer UI и Backend не должны зависеть от способа реализации бизнес-логики. Замена CRUD на FSM Engine не должна требовать изменения API Customer UI.

### 8. Новый документ

Создать новый документ `022 - Совместимость с будущим FSM Engine.md`:

```
# Совместимость Customer UI с будущим FSM Engine

## Цель
Подготовить Customer UI к внедрению FSM Engine без изменения
пользовательских сценариев и пользовательского интерфейса.

## Основные требования
1. Все действия проходят через Action Catalog.
2. Все изменения предметной области порождают Business Events.
3. Customer UI работает только с ViewModel.
4. Customer UI не анализирует status.
5. Customer UI отображает доступные Actions, а не вычисляет их.
6. Backend может использовать CRUD до внедрения FSM Engine.
7. После внедрения FSM Engine должны измениться только
   внутренние сервисы обработки Action.

## Не допускается
- логика UI на основании status;
- прямое изменение бизнес-сущностей из UI;
- зависимость ViewModel от реализации CRUD;
- изменение Action Catalog при внедрении FSM Engine;
- изменение Business Event Catalog при внедрении FSM Engine.
```

**Важно (из промпта):** не изменяй существующую архитектуру документов и не переписывай текст без необходимости. Выполни только перечисленные изменения, сохранив стиль и структуру документации.
