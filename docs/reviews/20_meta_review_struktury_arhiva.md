# Мета-ревью структуры архива greenmarket_docs (v1/v2/v3) и рекомендация сменить режим работы

Источник: https://chatgpt.com/s/t_6a4e4da3c99081919c08dc723983da43

> ⚠️ Это не очередной документ серии ТЗ, а отдельная реплика о состоянии всего архива `greenmarket_docs/` (после того, как ему показали структуру v1/v2/v3). Прямого номера ТЗ не имеет.

**Реплика чата:** проверил структуру архива. Приятно удивила — видно, что документация уже начинает превращаться в полноценную проектную базу знаний.

## Что уже хорошо

Есть три версии документации:

- **v1** — эволюция идеи и первоначальная архитектура.
- **v2** — модель БД, каталог, API, публикация, MVP.
- **v3** — полностью отдельный пакет Customer UI.

Это правильное разделение. История решений не потеряна.

## Что вызывает беспокойство

В v3 уже 21 документ (по внутренней нумерации ТЗ, без учёта пропусков — фактически 19 файлов, см. `README.md` этой папки). И это произошло всего за один день обсуждения.

Если продолжать в таком темпе, через месяц будет уже 70–100 документов. Разработчики перестанут понимать: «Что обязательно читать?»

## Предложение по структуре

Сейчас изменить не содержание, а структуру. Например:

```
greenmarket_docs/
  00_Project/
  01_Product/
  02_Domain/
  03_Customer_UI/
  04_Seller_UI/
  05_Admin_UI/
  06_Backend/
  07_Database/
  08_API/
  09_Algorithms/
  10_Platform_Integration/
  99_ADR/
```

Тогда сегодняшние документы автоматически разложатся, например:

**Customer_UI:** Главный экран; Карточка продавца; Карточка товара; User Flow; FSM; ViewModel; NFR.

**Domain:** Предметная модель; Глоссарий; Информационная модель; Бизнес-правила.

**Algorithms:** Purchase Optimizer; Варианты покупки.

**Platform Integration:** Actions; Events; Контракт UI-Backend.

## Проблема с нумерацией

Документы сейчас идут по номерам (001, 002, 003…), но номер перестанет что-либо означать — через месяц искать станет тяжело. Номер можно оставить только как внутренний идентификатор, а не как основу именования/навигации.

## Самое важное: предложение прекратить создавать новые документы

Не потому что идей нет, а потому что сейчас наступает следующий этап проекта — нужно начать **наполнять существующие документы**, а не создавать новые.

Например, ТЗ-001 «Главный экран» сейчас — около 2–3 страниц. Ожидается, что после появления первого дизайна он станет документом на 15–20 страниц: состояния; все элементы; все действия; ошибки; жесты; разрешения; офлайн; пустые состояния; доступность. То же касается карточки продавца, товара и остальных документов.

## Смена режима работы

До сегодняшнего дня работа шла **вширь** — создавались новые документы. Предлагается работать **вглубь** — брать один документ и доводить его до состояния, когда разработчик может реализовать его практически без дополнительных вопросов.

Общая архитектура Customer UI уже достаточно хорошо сформирована. Дальнейшее развитие проекта даст больше пользы через детализацию существующих ТЗ, а не увеличение их количества — это также облегчит работу разработчикам GreenMarket: вместо десятков новых файлов они будут получать всё более полные и самодостаточные спецификации по своим направлениям.
